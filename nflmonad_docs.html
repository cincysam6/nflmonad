Overview

nflmonad is a self-contained NFL analytics platform. It automatically downloads NFL data every day, cleans and transforms it through multiple layers, and produces model-ready datasets for betting and projection workflows.

Think of it as a pipeline that runs on autopilot — data flows in from public NFL data sources, gets cleaned up and enriched, and lands in structured tables ready for modeling. You don't need to manually download spreadsheets or wrangle data; the platform handles all of that and then surfaces clean, well-labeled features for you to build models on top of.

What problem does this solve?

Building a good NFL betting model requires dozens of datasets — play-by-play data, injury reports, snap counts, roster changes, odds lines, weather — all joined together consistently and without "cheating" by using information that wouldn't have been available before the game was played. Setting that up from scratch is hundreds of hours of work. nflmonad does all of it for you.

For Modeling

Pre-built, leakage-safe feature tables covering team strength, injury burden, rest, schedule, and betting lines — ready to drop into XGBoost, regression, or any other model.

For Betting Analysis

Closing line value calculations, ATS tracking, over/under performance, and model vs. market comparison built into the validation suite.

Technology Stack

  • R — the primary language for data ingestion, transformation, and modeling
  • nflverse / nflreadr — free, authoritative NFL data (play-by-play, schedules, rosters, injuries, snap counts)
  • DuckDB — fast in-process analytical SQL engine; runs locally with no server needed
  • Parquet — columnar file format for storage; think of it as "smart CSV" that's 10x faster to query
  • GitHub Actions — runs the pipeline automatically on a schedule (no manual effort required in-season)
  • XGBoost — the modeling engine for game prediction

Why this approach?

Each layer has a specific job and a clear contract. If something goes wrong — say a new season has a column name change in the source data — you fix it in one place (the staging layer) and everything downstream rebuilds automatically. You never have messy join logic scattered across model notebooks.

Each layer is stored as Parquet files partitioned by season. This means querying 2024 data only reads 2024 files — it's fast even on a laptop, and the structure mirrors what you'd have in a cloud warehouse like BigQuery or Snowflake.

Silver Layer — Staging

Staging tables standardize the raw data: types are cast consistently, team abbreviations are harmonized (e.g. OAK → LV, STL → LA), and the grain (what each row represents) is made explicit and enforced. Think of this as "cleaned raw data."

Design Rule
Staging tables never add business logic or derived features. They only clean, type, and rename. If you add a new column here, it must come directly from the raw source.

Intermediate Layer

The intermediate layer is where feature engineering happens. These tables compute the rolling metrics, lags, and derived signals that models actually use. All metrics here are lag-safe by design — they only incorporate data from games that have already been played before the current game.

⚠️ Critical: The Lag Rule
Every rolling metric in this layer uses ROWS BETWEEN UNBOUNDED PRECEDING AND 1 PRECEDING in its SQL window function. This means the current game is never included in its own features. A Week 10 game only "knows" about Weeks 1–9.

Leakage Rules

Data leakage is the cardinal sin of sports modeling. It happens when your model is accidentally trained on information that wouldn't have been available before the game was played. A model with leakage will look spectacular in backtest and fall apart on real predictions.

nflmonad enforces leakage prevention at every layer. Here are the key rules:

Rule 1: All rolling metrics are lagged

Every window function in the SQL uses ROWS BETWEEN UNBOUNDED PRECEDING AND 1 PRECEDING. This means when computing "Kansas City's EPA per play entering Week 10", we only include Weeks 1–9. Week 10's game is never part of its own features.

Rule 2: Targets are always separate from features

Columns like home_score, home_margin, and home_win_flag are NULL for future games and only populated after the game is played. The model cannot accidentally see the answer.

Rule 3: Temporal cross-validation only

Never randomly split data into train/test by row. Always split by season. A model trained on 2015–2021 data should be tested on 2022+ data — never tested on a random 20% of 2015–2021 games mixed in.

Rule 4: Opening vs. closing odds

Opening lines are available before the game — they're safe as model features. Closing lines represent the final market consensus and are only available right before kickoff. Use closing lines only as targets for closing line value (CLV) analysis, never as input features for a pregame model.

⚠️ Common Trap: Injury Reports
The injury report used as a feature should be the one filed before the game — the final Friday or Saturday report. Using the actual "Did Not Participate" outcome from the game itself would be leakage. int_injury_team_impact handles this correctly.

Feature Reference

What is EPA?

Expected Points Added (EPA) is the most important metric in modern NFL analytics. It answers: "How much did this play change the expected point outcome of the game?" A 4-yard gain on 3rd and 3 has high positive EPA (converted the down). A 4-yard gain on 3rd and 10 has negative EPA (failed to convert). EPA is significantly better than raw yards at measuring team quality.

Understanding the three rolling windows

Season-to-date (_std)

All prior games this season. Most stable, but slow to react to major changes (injuries, coaching adjustments). The primary signal for game modeling.

Last 5 / Last 3 (_l5, _l3)

Recent form. More volatile but captures hot streaks, coaching changes, and post-injury recovery. Less reliable early in the season.

The Blended Metric
epa_pp_blended = 0.70 × current_season_epa + 0.30 × prior_season_epa. This solves the Week 1–3 problem where teams have almost no current-season data. The prior season acts as a regularizing prior — a Bayesian concept where you start with a reasonable belief about a team's quality and update it as new evidence comes in.

How the Models Work

The modeling layer consists of three R files that work in sequence: training, simulation, and validation. You can run them independently or chain them together with a single command.

Three simultaneous models

Every time you train, three models are built at once — they share the same features but predict different things:

Spread Model

Predicts home_margin (home score minus away score). The output is a continuous number like "home wins by 4.2 points." Used for against-the-spread analysis.

Total Model

Predicts total_points (combined final score). Used for over/under analysis. Weather and pace features are most important here.

Win Probability Model

Predicts P(home wins) as a probability 0–1. Built as a binary classifier. Complements the spread model — they should agree on direction but may differ on certainty.

Why three?

Cross-checking models against each other catches errors. If the spread model says "home wins by 6" but the win probability model says "50/50", something is off in one of the models.

XGBoost — why this algorithm?

XGBoost is a gradient-boosted decision tree algorithm. It's the right choice for NFL game prediction because: it handles missing values natively (games early in the season have thin rolling windows), it works well on small datasets (~1,100 training games is small by ML standards), it automatically captures non-linear relationships (e.g. the effect of rest might only matter when combined with a difficult opponent), and it provides feature importance scores that tell you what's actually driving predictions.

The conservative hyperparameter choice

NFL data is small. With only ~1,100 training games, overfitting is a serious risk. The models use conservative settings: min_child_weight=10 (each leaf needs at least 10 games), lambda=2.0 (L2 regularization), and max_depth=4 (shallow trees). Early stopping terminates training when validation score stops improving.

Monte Carlo Simulation

The simulation layer is what separates a prediction from a betting decision. Here's the core idea in plain language:

Your XGBoost model says "the home team wins by 4.2 points on average." But NFL games don't happen on average — they're individual events with enormous variance. The standard deviation of NFL game margins is about 13.5 points. That means a team predicted to win by 4.2 actually wins the game only about 58% of the time.

Why simulate instead of just using the model probabilities?

The win probability model gives you P(home wins) directly. But the simulation approach has advantages: it lets you derive cover probabilities (not just win probabilities) against any spread line, it gives you the full distribution of outcomes (not just the mean), and it lets you run scenarios like "what if the line moves from -3 to -4.5?"

Expected Value (EV) calculation

Once you have a cover probability, you can calculate expected value:

# At -110 juice (standard), you risk $110 to win $100
# Breakeven win rate: 110/210 = 52.38%

# If simulation says p_home_cover = 0.56:
EV = (0.56 × $100) - (0.44 × $110) = $56 - $48.40 = +$7.60 per $110 bet
ROI = 7.60 / 110 = 6.9% edge

A positive EV means the bet has long-run positive expected return. Real edges in NFL betting are typically in the 2–5% range — anything above 10% should be treated skeptically (likely a data error or overfitting).

Validation Suite

The validation suite in R/models/validate_model.R produces seven categories of performance reports. Run run_model_pipeline("validate") to generate all of them.

1. Accuracy Metrics

  • RMSE (Root Mean Squared Error) — average magnitude of prediction error, in points. A spread RMSE of 10–12 is typical for NFL; below 10 is excellent.
  • MAE (Mean Absolute Error) — similar to RMSE but less sensitive to outliers (blowouts).
  • R² — how much of the variance in outcomes the model explains. NFL game outcomes are highly variable; an R² of 0.15–0.25 is realistic.
  • Bias — is the model systematically over- or under-predicting? Should be near zero.

2. Betting Performance

  • ATS record — wins and losses against the spread across the test set
  • O/U record — wins and losses on over/under bets
  • ROI — return on investment at standard -110 juice. Breakeven is 52.38%.
  • Edge-filtered ROI — same metrics but only for games where the model had a 5%+ edge, to test whether selectivity improves performance

3. Calibration

Win probability calibration asks: "When my model says 60% probability, do the home teams win 60% of the time?" A perfectly calibrated model's points lie on the diagonal line. Deviation from the diagonal means the model is overconfident or underconfident at certain probability levels.

4. Segment Analysis

Breaks down model accuracy by game type: divisional vs. non-divisional, primetime vs. regular, dome vs. outdoor, etc. This tells you if the model has systematic blind spots (e.g., consistently poor at predicting divisional games).

5. Season-by-Season

Shows RMSE, MAE, R², and ATS record for each individual season. A good model should be consistent across seasons. If 2019 is dramatically different from 2020, investigate why — there may be a structural change in the data (COVID, rule changes) or a regime change the model doesn't handle.

6. Closing Line Value (CLV)

CLV analysis asks: "Does the model's prediction align with where the betting line moved?" If the line opened at -3 and moved to -4.5, sharp bettors pushed it to -4.5 — they had information the market hadn't priced in. If your model also predicted -4 or -5, you had the same information. High CLV agreement (consistently above 52%) suggests the model has real signal.

Function Reference

Pipeline Functions (pipeline.R)

run_backfill(cfg, force_full = FALSE)
Downloads all historical data from historical_start through current_season and runs full transforms. Run once during initial setup. Takes 30–90 minutes depending on season range.
cfgConfig list from load_config()
force_fullIf TRUE, re-downloads all seasons even if Parquet files already exist
run_daily_refresh(cfg)
Downloads current season data only and runs incremental transforms. Designed to run nightly via GitHub Actions. Completes in 5–15 minutes.

Ingestion Functions (R/ingest/)

ingest_pbp(seasons, cfg, force_full)
Downloads play-by-play data. The largest and slowest download — ~2-3 minutes per season. PBP is the source for EPA calculations.
ingest_rosters_weekly(seasons, cfg, force_full)
Downloads weekly roster snapshots using nflreadr::load_rosters_weekly(). Gives you the active 53-man roster state for each week, which is essential for tracking when players joined/left a team.
ingest_participation(seasons, cfg, force_full)
Downloads play-level participation data — which specific players were on the field for each play. Available from 2016 onward. Large dataset but enables true snap-level analysis.

Transform Functions (R/transform/run_transforms.R)

run_transforms(cfg, incremental = FALSE)
Orchestrates the full SQL pipeline: registers raw Parquet views, runs staging SQL, runs intermediate SQL, runs mart SQL. This is the function that moves data through all three layers.
incrementalIf TRUE, only rebuilds current season. If FALSE, rebuilds all seasons from scratch.

Model Functions (R/models/)

train_game_models(cfg, save_dir)
Trains all three XGBoost models (spread, total, win probability) on mart_backtest_game with temporal cross-validation. Saves model artifacts to models/game/.
simulate_games(predictions, margin_sd, total_sd, n_sims)
Runs Monte Carlo simulation for all games in the predictions data frame. Returns probability distributions, cover probabilities, and EV calculations.
predictionsOutput from predict_games()
margin_sdStandard deviation of the simulation noise. Default 13.5 pts (calibrated from historical residuals).
n_simsNumber of Monte Carlo draws per game. Default 10,000.
run_validation_suite(sim_df, report_dir, cv_split_filter)
Runs all seven validation categories (accuracy, betting performance, calibration, CLV, segment analysis, season-by-season) and saves results to reports/model_validation/.
calibrate_residuals(predictions)
Computes empirical residual standard deviation from the test set. Run this after training to update the simulation noise parameters. Output tells you the true spread and total standard deviations for your model specifically.
calc_ev(p_win, american_odds, stake = 100)
Calculates expected value for a bet given a win probability and American odds. Returns the expected profit and ROI per bet.
p_winYour model's estimated probability of winning the bet (0–1)
american_oddsThe line in American format (e.g. -110, +150)

Reads config/config.yml and returns a named list. All pipeline functions take cfg as their first argument — always start with this.
db_connect(cfg, read_only = FALSE)
Opens a DuckDB connection to db/nfl_analytics.duckdb. Always pair with on.exit(db_disconnect(con), add = TRUE) to prevent file locks.

What "closing line" means for modeling

The closing line is the final market consensus before kickoff. It incorporates all public information — injury reports, weather, sharp betting action, public sentiment. It's the single best publicly available predictor of game outcomes (better than most models). Using it as a feature is valid for pregame prediction models. Using the closing line as your target (measuring whether your model predicted the final market correctly) is called Closing Line Value and is the gold standard for evaluating model quality.

FAQ

Why does the pipeline build team_stats from raw_pbp instead of downloading it?

nflverse doesn't provide a pre-built series conversion rate table. The platform computes it using nflfastR::calculate_series_conversion_rates() applied to the raw play-by-play data that's already been downloaded. This adds a few minutes to transforms but saves a separate API call.

Why are there only ~1,108 rows in mart_backtest_game if there are 10 seasons of data?

The backtest mart filters to completed regular season games where both teams have played at least one prior game. This excludes Week 1 (no prior-game features exist), preseason, and postseason. A 17-game regular season × 32 teams / 2 teams per game = 272 games per season × 10 seasons = 2,720, minus the Week 1 filter ≈ 1,108. That's correct.

The model shows 54% ATS — is that good?

Yes. Breakeven at standard -110 juice is 52.38%. Any sustained edge above 52.4% is profitable over the long run. 54% represents about a 3% ROI per bet. Sharp sportsbooks would limit you at those numbers. Be appropriately skeptical — make sure you're evaluating on a truly held-out test set (2022+) and not on data that was touched during training.

Can I use this for in-game betting?

Not out of the box. The current platform is built for pregame prediction. In-game models require real-time win probability using live game state (score, time remaining, possession) — a fundamentally different problem. The play-by-play data is all here for building it, but it would require a separate modeling pipeline.

How do I add a new feature?

Four steps: (1) Add the raw column in the appropriate staging SQL, (2) add a derived calculation in the relevant intermediate SQL, (3) add it to the mart SQL select list, (4) add its column name to .GAME_FEATURES or .ENGINEERED_FEATURES in train_game_model.R. Re-run run_transforms() and then retrain.

Why is R used instead of Python?

nflverse is the authoritative NFL data ecosystem and is entirely R-based. The R packages nflreadr, nflfastR, and nflplotR have no equivalent Python alternatives with the same coverage or maintenance. The Parquet files produced by the pipeline are format-agnostic — you can consume them in Python with pandas.read_parquet() or polars.read_parquet() if you prefer to model in Python.

How do I update the platform to a new NFL season?

In config/config.yml, change current_season: 2024 to current_season: 2025. That's it. The pipeline handles everything else automatically — the model's cv_split will continue to assign the newest season to "test" based on the existing time-based logic.
